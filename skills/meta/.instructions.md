## Creating / Updating Specifications (`create-spec` / `update-spec`)

Use when: writing a new spec, iterating on an existing spec after testing reveals gaps.

**Output location:** `docs/ctrl-specific/<VENDOR>/spec/<name>.md`
Naming: `<scope>-<component>.md` — e.g., `basic-downloader.md`, `arcwelding-plugin.md`

### Spec template (frontmatter + required sections)

```md
---
title: [Title]
version: 1.0
date_created: YYYY-MM-DD
last_updated: YYYY-MM-DD
vendor: KAWASAKI
tags: [downloader, arc-welding, ...]
---

# Introduction
[What this spec defines and the goal it achieves.]

## 1. Purpose & Scope
## 2. Definitions
| Term | Definition |
|------|-----------|
## 3. Requirements & Constraints
- **REQ-001**: ...
- **CON-001**: ...
## 4. E2 Object Mapping
| E2 Source | Robot Output | Notes |
|-----------|-------------|-------|
## 5. Output Format
[Annotated examples from golden files]
## 6. Acceptance Criteria
- **AC-001**: Output matches `golden/<FILE>.as`
## 7. Edge Cases & Notes
## 8. Related Specs
```

### Update rules
- Preserve existing requirement IDs; append new ones with next sequential number
- Strike through removed requirements: `~~REQ-003~~`
- Update `last_updated` and bump `version` for significant changes
- Add a changelog section after more than two updates

## Creating / Updating Implementation Plans (`create-plan` / `update-plan`)

Use when: planning a multi-phase feature, mapping a workshop roadmap, tracking progress.

**Output location:** `docs/ctrl-specific/<VENDOR>/plan/<name>.md`
Naming: `<purpose>-<component>.md` — e.g., `feature-kawasaki-downloader.md`

### Plan template

```md
---
goal: [Goal statement]
version: 1.0
date_created: YYYY-MM-DD
last_updated: YYYY-MM-DD
owner: [Person/team]
status: 'Planned'
tags: [feature, upgrade, roadmap]
---
# Introduction
![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

## 1. Requirements & Constraints
- **REQ-001**: ...
- **CON-001**: Must not break existing golden file output

## 2. Implementation Phases

### Phase 1: [Name]
- GOAL-001: [Measurable goal]

| Task | Description | Status | Date |
|------|-------------|--------|------|
| TASK-001 | [Task with file paths] | | |

**Acceptance**: [How to verify — e.g., "golden/PG06_BASIC.as matches"]

## 3. Alternatives Considered
- **ALT-001**: [Alternative and rejection reason]
## 4. Dependencies
## 5. Files Affected
## 6. Testing
## 7. Risks & Assumptions
```

### Update rules
- Mark completed tasks: `| TASK-001 | Description | ✅ | YYYY-MM-DD |`
- Append new tasks with next sequential ID
- Update `status` in frontmatter on phase transitions: `Planned → In progress → Completed`
- Never delete completed tasks

## Technical Documentation (`documentation-writer`)

Use when: writing workshop guides, how-to docs, reference tables, or explanations.

Follow the **Diátaxis framework** — four distinct document types:

| Type | Purpose | Use For |
|------|---------|---------|
| **Tutorial** | Learning — guide a newcomer to a successful outcome | Workshop exercises, onboarding |
| **How-to Guide** | Problem-solving — steps to solve a specific task | "How to add arc welding support" |
| **Reference** | Information — technical descriptions for lookup | API tables, attribute lists, command syntax |
| **Explanation** | Understanding — clarify WHY, not just WHAT | Architecture rationale, design decisions |

**Workflow:** Clarify type + audience + goal → propose structure → generate content.

**Style for this repo:**
- Use E2 API terminology consistently (attributes, events, operations, etc.)
- Reference golden files and tree dumps when explaining output format
- 3-space Python indentation in code examples
- Keep docs actionable; avoid "wall of text"

## Creating New Skills (`make-skill`)

Use when: scaffolding a new controller skill (e.g., FANUC LS language, YASKAWA INFORM), domain skill, or meta skill.

```
skills/<skill-name>/
└── SKILL.md          # Required
```

### SKILL.md frontmatter rules

```yaml
---
name: <skill-name>           # lowercase, hyphens, matches folder name
description: '<WHAT it does. Use when <TRIGGERS, scenarios, keywords>.'
---
```

**CRITICAL:** The `description` is the primary auto-discovery mechanism. Include:
1. What the skill does
2. When to trigger it (exact phrases users might say)
3. Domain keywords (robot command names, language syntax terms)

Optional subdirectories: `references/` (docs the agent reads), `templates/` (starter code to extend).

## Converting Plaintext to Markdown (`convert-plaintext-to-md`)

Use when: converting a PDF-extracted controller manual, raw text dump, or unformatted reference into structured markdown.

**Guidelines for controller manuals:**
- Use `## Command Name` for each command/instruction
- Put syntax in fenced code blocks with language hint (e.g., ` ```as ` for Kawasaki AS)
- Format parameter tables as markdown tables
- Preserve all technical content exactly — never change data
- Add a table of contents for documents over 100 lines
- Use `> [!NOTE]` callouts for important warnings
- Target: **GitHub-flavored markdown (GFM)**
